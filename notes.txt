difference between void  & never

void-The function finishes, but returns nothing (undefined).
never-The function never reaches the end.
.In the context of a function return type, never tells TypeScript: "This function will never finish its execution normally."
main roles of never:
1. Functions that throw errors
As in your code, the function starts, but it reaches a throw statement before it can reach the end or return a value. The execution path is "broken."
2. Infinite Loops
If a function contains a while(true) loop that has no break or return, it also returns never because it stays "trapped" in the function forever.
3. Exhaustiveness Checking (Exhaustive Pattern Matching)
This is the most "Intermediate" use case. It ensures you have handled every possible case in a switch or if/else block. If you handle every case of a Union type, 
the remaining possibility should be never.


Property Initialization Methods :

1.Direct initialisation:
Ex:-
class Car {
  brand: string = "Toyota"; // Initialized directly
  color: string;
  // ...
}
use cases:
.for Fixed constants or "safe" starting values that don't depend on external input.

2.constructor initialization:
ex code:-
class Car {
  brand: string;
  color: string;
  constructor(color: string) {
    this.brand = "Toyota";
    this.color = color; // Initialized in the constructor
  }
}

use cases:
.Required data that varies for every instance.ex:An Employee class where you 
must provide a unique id and name at the moment the object is created.

3.Optional Properties:
ex code:-
class Car {
  brand?: string; // Optional, can be undefined initially
  // ...
}

use case:
. Data that is truly "extra" or might not be available yet.
ex: A Profile class where a middleName or phoneNumber is not required for everyone.

4.Parameter Properties (Shorthand)
ex code:
class Car {
  // TypeScript automatically creates and initializes the 'color' property
  constructor(public color: string) {}
}

usecase:
.Reducing "boilerplate" code when you have many properties.
ex:A Point class with x, y, and z coordinates. Instead of declaring them at the
 top and assigning them inside the constructor, you do both in one line.

5.Definite Assignment Assertion:
ex code:
class Car {
  brand!: string; // Asserts it will be assigned later
  // ...
}
use case:
.Using ! tells the compiler that the property will be assigned a value
 later (e.g., via an external library or a method), even if it's not in the constructor


when a Constructor is Needed?
1.Strict Property Initialization: If the strictPropertyInitialization flag is enabled in your tsconfig.json, TypeScript will throw an error if a property is neither initialized at declaration nor in the constructor.
2.External Input: If you need to set property values based on arguments passed when using new ClassName(arg), you must use a constructor.
3.Readonly Properties: readonly fields must be initialized either at their declaration or inside the constructor.
4.Inheritance: If a class extends a base class and has its own constructor, it must call super() to initialize the parent.


//Type Narrowing:

You can say a code block is performing type narrowing if:
Before: The variable could be multiple things (a Union type).
The Trigger: You use a conditional check (like if or switch).
After: Inside that block, TypeScript "knows" exactly what the type is and lets you use methods specific to that type.

4 Common Ways to Narrow a Type:
    1.The "typeof" Guard (Primitive Narrowing):
    ex code:
    function format(input: string | number) {
      // At this point, 'input' is string | number
      if (typeof input === "string") {
        // NARROWING HAPPENED HERE
        return input.toUpperCase(); // Safe! TypeScript knows it's a string.
      }
      return input.toFixed(2); // Also narrowed! If it's not a string, it must be a number.
    }

    2.The Truthiness Check:
    function printName(name?: string) {
  if (name) { 
    // NARROWING: 'name' is now 'string', not 'string | undefined'
    console.log(name.length);
  }
}    
     
      3.Equality Narrowing
      Using === or !== to compare values.
      ex code:
      function check(x: string | number, y: string | boolean) {
  if (x === y) {
    // NARROWING: For x to equal y, they both MUST be 'string'.
    // x is now 'string' inside this block.
  }
}

     4.The in Operator (Object Narrowing)
     ex code:
     type Fish = { swim: () => void };
type Bird = { fly: () => void };

function move(animal: Fish | Bird) {
  if ("swim" in animal) {
    // NARROWING: TypeScript knows 'animal' is a Fish.
    animal.swim();
  }
}

//Getters and Setters

why we use them?
.Consistency
.Side Effects:to trigger actions like logging or updating a ui,every time a specific value changes.

when we use them:
1.for Data Validation
2.for Computed Properties
3.for Encapsulation(Read-only access)
4. Debugging and Logging
If you have a bug where a value is changing unexpectedly, you can put a console.log inside a setter. 
Every time that property is modified anywhere in your app, it will trigger the log, telling you exactly when the change happened.

Important Syntax Tip
In TypeScript, you usually prefix the actual private variable with an underscore (e.g., _name) 
to avoid a naming collision with the getter/setter names.

class Account {
  private _balance: number = 0;

  get balance() { return this._balance; } // Read-only public access
}

Combination      |             Result
Get + Set        |      Standard read/write property with added logic.
Get only         |      Read-only property (very common for IDs or totals).
Set only         |      Write-only property (rare, but used for things like "password" updates).
